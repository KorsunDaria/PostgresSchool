# Изучение исходного кода PostgreSQL: структура репозитория, запуск главного процесса и механизм расширений

## Введение

PostgreSQL — один из самых зрелых и хорошо документированных проектов с открытым исходным кодом среди СУБД. Несмотря на внушительный объём кода (несколько миллионов строк на C), репозиторий организован достаточно логично, а сама СУБД спроектирована так, чтобы значительную часть функциональности можно было добавлять снаружи — через **расширения (extensions)**, не трогая ядро.

Цель этого доклада — дать базовое, но целостное понимание четырёх вещей:

1. как устроена структура репозитория PostgreSQL;
2. как запускается и работает главный (управляющий) процесс — **postmaster**;
3. как устроено расширение изнутри — на примере штатного расширения **pg_stat_statements**;
4. как расширение подключается к серверу как разделяемая библиотека — от компиляции `.so`-файла до загрузки его в память сервера.

Материал основан на официальной документации PostgreSQL, исходном коде проекта (репозиторий `postgres/postgres`) и ряде авторитетных технических разборов внутреннего устройства (EnterpriseDB, Highgo Software, DeepWiki-разбор архитектуры, wiki-страницы разработчиков PostgreSQL). Список источников — в конце документа.

## 1. Структура репозитория PostgreSQL

Исходный код PostgreSQL хранится в Git-репозитории, зеркало которого доступно на GitHub (`postgres/postgres`). Официальный мастер-репозиторий поддерживается Anthropic... нет, поддерживается сообществом PostgreSQL Global Development Group, а GitHub — лишь read-only зеркало: пул-реквесты туда не принимаются, патчи отправляются через почтовую рассылку по специальной процедуре (`Submitting a Patch`).

Корень репозитория содержит сравнительно немного директорий верхнего уровня, самая важная из которых — `src`. Внутри `src` код разделён по принципу "frontend / backend / общее", что сделано намеренно: клиентские утилиты не должны тянуть за собой серверные заголовки и наоборот.

Основные поддиректории:

- **`src/backend`** — сердце СУБД, код серверного процесса. Внутри — множество подсистем:
  - `access/` — методы доступа к данным: реализация B-tree, GiST, GIN, BRIN, hash-индексов, heap-хранилища, транзакционного менеджера (`access/transam`);
  - `catalog/` — работа с системным каталогом (pg_class, pg_attribute и т.д.);
  - `commands/` — обработчики SQL-команд (CREATE TABLE, ALTER, VACUUM и др.);
  - `executor/` — исполнитель плана запроса;
  - `optimizer/` — планировщик/оптимизатор запросов;
  - `parser/` — синтаксический и семантический разбор SQL;
  - `postmaster/` — код главного процесса и вспомогательных фоновых процессов (именно тут лежит `postmaster.c`, к которому мы вернёмся в разделе 2);
  - `storage/` — буферный кэш, менеджер блокировок, WAL, работа с файлами;
  - `tcop/` ("traffic cop") — верхнеуровневый цикл обработки запроса одним backend-процессом (`postgres.c`);
  - `utils/` — вспомогательные утилиты: управление памятью (`utils/mmgr`), GUC-параметры (`utils/misc/guc.c`), кэши каталога и т.д.

  Почти в каждой такой поддиректории лежит файл `README`, кратко описывающий назначение модуля — это удобная отправная точка при первом знакомстве с конкретной подсистемой.

- **`src/include`** — заголовочные файлы (`.h`), зеркалирующие структуру `src/backend` и являющиеся публичным (в рамках сервера) API подсистем. Именно эти заголовки устанавливаются как `server`-заголовки и используются при сборке сторонних расширений.

- **`src/bin`** — исходный код клиентских и серверных бинарных утилит: `psql`, `pg_dump`, `pg_ctl`, `pg_basebackup`, `initdb` и др. Это фронтенд-код, который может собираться и работать без запущенного сервера.

- **`src/interfaces`** — клиентские библиотеки, прежде всего `libpq` — базовая C-библиотека для соединения с сервером, на которой строятся почти все остальные драйверы (psycopg, JDBC через отдельный проект и т.д. используют собственные реализации протокола, но идеологически ориентируются на libpq).

- **`src/fe_utils`** — код, общий для разных frontend-утилит, но не являющийся частью backend.

- **`src/port`** — платформенно-зависимые реализации системных функций, обеспечивающие переносимость между Linux, macOS, Windows, *BSD и др.

- **`src/test`** — регрессионные и другие автоматические тесты сервера.

- **`src/pl`** — процедурные языки (PL/pgSQL, PL/Perl, PL/Python, PL/Tcl).

- **`src/timezone`** — поддержка часовых поясов.

- **`contrib`** — отдельная директория верхнего уровня (не внутри `src`), в которой находятся необязательные модули и расширения, поставляемые вместе с исходным кодом PostgreSQL, но не входящие по умолчанию в собранное ядро. Именно здесь лежит `pg_stat_statements`, который мы разберём в разделе 3. По собственному признанию разработчиков, весь `contrib` служит одновременно и полезным набором готовых расширений, и справочным набором примеров того, как расширение вообще следует писать.

- **`doc`** — исходники документации в формате DocBook/SGML, из которых генерируется в том числе HTML-версия официальной документации на сайте postgresql.org.

Отдельно стоит отметить служебные инструменты для разработчиков: `src/tools/pgindent` (форматирование кода в едином стиле, применяемое ко всей кодовой базе перед каждым релизным циклом), `src/tools/editors` (настройки для vim/emacs под стиль PostgreSQL) и `src/template` — шаблоны для добавления поддержки новых операционных систем при портировании.

Практический совет для первого знакомства с деревом исходников: выполнить `tree -d -L 3 src` (или аналог), после чего целенаправленно открывать `README`-файлы интересующих подсистем — это быстрее, чем пытаться читать код "с нуля" без обзорного описания.

## 2. Как запускается главный процесс PostgreSQL (postmaster)

PostgreSQL использует классическую **процессную (не потоковую) модель**: на каждое клиентское соединение создаётся отдельный OS-процесс, а не поток внутри одного процесса. Это исторически осознанное архитектурное решение — авторы проекта долгое время считали, что кроссплатформенная поддержка потоков была недостаточно надёжной, а изоляция через отдельные процессы даёт более предсказуемую отказоустойчивость: падение обработчика одного соединения не должно ронять весь сервер.

### 2.1. Что такое postmaster

Когда администратор запускает сервер (обычно через `pg_ctl start`, который, в свою очередь, запускает бинарник `postgres`), стартует процесс, называемый **postmaster** — это тот же исполняемый файл `postgres`, просто запущенный в режиме родительского управляющего процесса. Postmaster:

- слушает TCP/IP-порт (по умолчанию 5432) и Unix-сокет;
- является родителем всех остальных процессов кластера;
- порождает (`fork`) отдельный **backend-процесс** на каждое новое клиентское соединение;
- запускает и следит за набором вспомогательных фоновых процессов;
- отвечает за инициализацию разделяемой памяти при старте и за корректное завершение/восстановление кластера при сбоях.

### 2.2. Путь выполнения при старте

Точка входа — функция `main()`, которая для случая запуска сервера (а не отдельной клиентской утилиты) передаёт управление в `PostmasterMain()` (файл `src/backend/postmaster/postmaster.c`). Внутри `PostmasterMain()` последовательно происходит:

1. **Разбор аргументов командной строки** и связанных с ними GUC-параметров (`postgresql.conf`, аргументы `-D`, `-p` и т.п.).
2. **Проверки каталога данных (`PGDATA`)**: наличие файла блокировки, файла управления кластером (`pg_control`), создание собственного lock-файла с PID постмастера — это гарантирует, что с одним каталогом данных одновременно работает только один сервер.
3. **Загрузка библиотек, указанных в `shared_preload_libraries`**, и вычисление объёма разделяемой памяти, необходимого как самому ядру, так и загруженным библиотекам/расширениям (это ключевой момент, к которому мы вернёмся в разделе 4 — многие расширения обязаны быть загружены именно на этом шаге, чтобы успеть зарезервировать себе разделяемую память).
4. **Выделение сегмента разделяемой памяти** нужного размера (буферный кэш, WAL-буферы, таблицы блокировок, массив `PGPROC` и т.д.) функцией `PGSharedMemoryCreate()`.
5. Запуск процесса **Startup** (`StartupDataBase()`), который при необходимости выполняет восстановление после сбоя/применение WAL, переводя сервер в состояние `PM_STARTUP`.
6. Переход в главный бесконечный цикл — **`ServerLoop()`**.

### 2.3. ServerLoop и жизненный цикл процессов

`ServerLoop()` — это событийный цикл постмастера. В нём происходит:

- ожидание новых входящих соединений на сокете;
- при появлении соединения — вызов `BackendStartup()`, который через `fork()` (либо, на платформах без `fork()`, таких как Windows, — через отдельный механизм `EXEC_BACKEND` с `CreateProcess()`) порождает новый backend-процесс, наследующий доступ к уже созданной разделяемой памяти;
- периодическая проверка через `LaunchMissingBackgroundProcesses()`, все ли обязательные фоновые процессы живы, и перезапуск упавших при необходимости;
- обработка сигналов ОС (SIGHUP — перечитать `postgresql.conf`, SIGUSR1 — внутренние уведомления между процессами, сигналы завершения — см. ниже) через специальный обработчик, который выставляет флаги, а фактическая обработка происходит уже в теле `ServerLoop`.

Помимо backend-процессов (по одному на клиентское соединение), postmaster поддерживает набор служебных фоновых процессов, каждый из которых отвечает за свою задачу:

| Процесс | Назначение |
|---|---|
| **Checkpointer** | периодически создаёт контрольные точки, сбрасывает "грязные" страницы из буферного кэша на диск, позволяет переиспользовать старые сегменты WAL |
| **Background writer** | проактивно сбрасывает часть грязных страниц, чтобы у обычных backend-процессов было больше свободных буферов |
| **WAL writer** | сбрасывает журнал предзаписи (WAL) из буферов в файлы на диске |
| **Autovacuum launcher** | периодически запускает рабочие процессы autovacuum для очистки "мёртвых" версий строк |
| **Startup process** | применяет WAL при восстановлении после сбоя (на мастере) или непрерывно на реплике при потоковой репликации |
| **WAL sender / receiver** | передача WAL между основным сервером и репликами |
| **Logger (syslogger)** | сбор и запись серверных логов |

### 2.4. Завершение работы

Postmaster поддерживает три режима остановки, каждому из которых соответствует свой сигнал (это то, что стоит за флагом `-m` у `pg_ctl stop`):

- **smart** — сервер дожидается отключения всех клиентов и только потом завершается;
- **fast** (режим по умолчанию) — активные backend-процессы прерываются немедленно, но выполняется корректный checkpoint перед выходом;
- **immediate** — аварийное завершение без корректного шатдауна; при следующем старте потребуется восстановление после сбоя (crash recovery), как если бы произошёл реальный крах системы.

Когда падает (не штатно завершается) один из дочерних процессов, постмастер интерпретирует это как потенциальное повреждение разделяемой памяти и в целях безопасности завершает все остальные дочерние процессы, после чего инициирует восстановление — это стандартный защитный механизм архитектуры "один процесс — одно соединение".

## 3. Структура расширения на примере pg_stat_statements

`pg_stat_statements` — штатное расширение из `contrib/pg_stat_statements`, которое собирает статистику по выполняемым SQL-запросам (число вызовов, суммарное время планирования/выполнения, число прочитанных/записанных буферов и т.д.). Оно хорошо подходит как учебный пример, потому что демонстрирует практически весь типичный набор элементов расширения C-уровня.

### 3.1. Из чего состоит расширение

Каталог `contrib/pg_stat_statements` содержит:

- **`pg_stat_statements.c`** — основная логика на языке C;
- **`pg_stat_statements--<версия>.sql`** (и файлы `--<версия>--<версия>.sql` для миграций между версиями) — SQL-скрипт, который выполняется при `CREATE EXTENSION` и который создаёт в базе данных SQL-объекты: функции-обёртки (например, `pg_stat_statements_reset()`), представления (`pg_stat_statements`, `pg_stat_statements_info`);
- **`pg_stat_statements.control`** — control-файл расширения (см. ниже);
- **`Makefile`** — сборочный файл, использующий инфраструструктуру **PGXS** (см. раздел 4);
- директория с регрессионными тестами.

### 3.2. Control-файл

Файл `pg_stat_statements.control` — это метаданные расширения, которые читает команда `CREATE EXTENSION`. Обычно он содержит:

```
comment = 'track planning and execution statistics of all SQL statements executed'
default_version = '1.12'
module_pathname = '$libdir/pg_stat_statements'
relocatable = false
trusted = true
```

Ключевые поля: `default_version` — какая версия SQL-скрипта будет применена, если версия явно не указана; `module_pathname` — путь к разделяемой библиотеке (`$libdir` подставляется автоматически как каталог `PKGLIBDIR`); `relocatable` — можно ли переносить объекты расширения в другую схему после установки; `trusted` — может ли расширение быть установлено непривилегированным пользователем, обладающим правом `CREATE` в базе данных (актуально начиная с PostgreSQL 13).

### 3.3. Ключевые элементы C-кода

**`PG_MODULE_MAGIC`** — макрос, который должен присутствовать ровно один раз в каждой загружаемой библиотеке. Он вставляет в бинарный файл блок метаданных (версия протокола совместимости, размеры основных типов и т.п.), которые сервер сверяет при загрузке `.so`-файла, чтобы не допустить загрузку расширения, собранного под несовместимую версию/сборку сервера.

**`_PG_init(void)`** — специальная функция, которую сервер автоматически вызывает сразу после загрузки библиотеки в память (аналогично конструктору модуля). В `pg_stat_statements` эта функция:

- проверяет глобальный флаг `process_shared_preload_libraries_in_progress` — если расширение не было загружено через `shared_preload_libraries`, инициализация полезной нагрузки не выполняется (модуль формально "молчит", хотя SQL-объекты, созданные через `CREATE EXTENSION`, всё равно существуют);
- регистрирует собственные GUC-параметры расширения через `DefineCustomIntVariable()` и аналогичные функции (так в системе появляются параметры вида `pg_stat_statements.max`, `pg_stat_statements.track`);
- регистрирует callback для инициализации разделяемой памяти и хуки, встраивающие логику расширения в стандартный конвейер обработки запроса (в случае `pg_stat_statements` — хуки, вызываемые до и после планирования/выполнения запроса, чтобы замерить время и накопить статистику).

Симметричная (но необязательная) функция **`_PG_fini(void)`** может выполнять действия при выгрузке библиотеки — на практике для большинства расширений она не нужна, так как выгрузка `.so` в работающем сервере не поддерживается штатно.

**Хуки (hooks)** — это глобальные указатели на функции, которые ядро PostgreSQL вызывает в определённых точках конвейера выполнения запроса (например, `ExecutorStart_hook`, `planner_hook`, `post_parse_analyze_hook`). Расширение сохраняет предыдущее значение хука (на случай, если уже загружено другое расширение, использующее ту же точку) и подставляет вместо него свою функцию, которая обычно сначала выполняет полезную работу расширения, а затем вызывает сохранённый предыдущий обработчик. Именно через хуки `pg_stat_statements`, не изменяя ядро, "врезается" в цикл обработки каждого запроса и накапливает метрики.

### 3.4. Разделяемая память

Так как `pg_stat_statements` хранит агрегированную статистику, доступную из любого backend-процесса, ему нужна собственная область в общей разделяемой памяти сервера (обычные приватные структуры данных backend-процесса для этого не годятся — они не видны другим процессам). Именно поэтому расширение обязано быть указано в `shared_preload_libraries`: расчёт и резервирование размера этой области происходит на этапе старта постмастера (см. раздел 2.2, шаг 3), когда ещё можно безопасно выделить единый сегмент shared memory нужного размера. Если попытаться загрузить такое расширение позже (просто через `CREATE EXTENSION` без предварительной настройки `shared_preload_libraries`), пространство под него уже не будет зарезервировано, и сервер сообщит об ошибке при попытке использовать функциональность расширения.

## 4. Как подключаются библиотеки расширений

### 4.1. Два разных, но связанных действия

Важно разделять два процесса, которые часто путают:

1. **Компиляция и установка бинарной библиотеки** (`.so` на Linux/*BSD/macOS, `.dll` на Windows) в каталог `PKGLIBDIR`, а SQL- и control-файлов — в `SHAREDIR/extension`. Это делает функциональность физически доступной серверу.
2. **Регистрация расширения в конкретной базе данных** командой `CREATE EXTENSION имя;`, которая выполняет SQL-скрипт из шага выше и создаёт в текущей базе SQL-объекты (функции, типы, представления, операторы).

Библиотека может физически присутствовать на диске, но не быть загруженной ни в один процесс сервера, пока какая-то функция из неё не будет вызвана впервые (тогда backend-процесс сам динамически подгрузит `.so` через системный вызов, аналогичный `dlopen`) — это называется **обычной динамической загрузкой** и подходит для большинства простых расширений (например, реализующих только новые функции или типы данных, без собственной разделяемой памяти и без хуков, требующих раннего старта).

### 4.2. Когда обязательна предзагрузка через shared_preload_libraries

Как показано на примере `pg_stat_statements`, некоторым расширениям обычной динамической загрузки недостаточно — им нужно:

- зарезервировать область разделяемой памяти (доступную всем backend-процессам);
- зарегистрировать фоновый рабочий процесс (background worker);
- встроиться в хуки, которые должны быть установлены ещё до того, как к серверу подключится первый клиент.

Для этого используется параметр конфигурации **`shared_preload_libraries`** (устанавливается в `postgresql.conf`, требует перезапуска сервера, так как это `PGC_POSTMASTER`-параметр, а не параметр уровня сессии):

```
shared_preload_libraries = 'pg_stat_statements'
```

Список библиотек разбирается функцией `process_shared_preload_libraries()` (`src/backend/utils/init/miscinit.c`), которая последовательно грузит каждую библиотеку из списка ещё во время `PostmasterMain()`, до основного цикла `ServerLoop()`, и для каждой вызывает её `_PG_init()`.

Есть также похожие, но более узкие по области действия параметры:

- **`local_preload_libraries`** — библиотеки, загружаемые только для конкретной сессии (после аутентификации, по желанию непривилегированного пользователя, без перезапуска сервера) — полезно для отладочных/трассировочных расширений;
- **`session_preload_libraries`** — аналогично, но задаётся администратором и применяется ко всем новым сессиям без явного действия пользователя.

### 4.3. Сборка расширения: PGXS

Чтобы не заставлять авторов сторонних расширений копировать в свой Makefile десятки строк флагов компиляции, PostgreSQL поставляет инфраструктуру **PGXS** (PostgreSQL Extension Building Infrastructure). Утилита `pg_config` показывает, среди прочего, путь `PGXS`:

```
$ pg_config --pgxs
/usr/lib/postgresql/16/lib/pgxs/src/makefiles/pgxs.mk
```

Типичный минимальный `Makefile` стороннего расширения выглядит так:

```makefile
MODULES = my_extension
EXTENSION = my_extension
DATA = my_extension--1.0.sql
PGFILEDEDEP = pg_config

PG_CONFIG = pg_config
PGXS := $(shell $(PG_CONFIG) --pgxs)
include $(PGXS)
```

После `make && make install` библиотека и сопутствующие файлы автоматически копируются в правильные каталоги (`PKGLIBDIR`, `SHAREDIR/extension`), взятые из той же установки PostgreSQL, на которую указывает `pg_config` — это гарантирует совместимость версии заголовков (`INCLUDEDIR-SERVER`) и версии сервера, для которого собирается расширение.

### 4.4. Итоговый путь от кода до работающего расширения

1. Написать `.c`-файл с `PG_MODULE_MAGIC`, функциями и (при необходимости) `_PG_init`/хуками; описать SQL-интерфейс в `.sql`-скрипте; создать `.control`-файл.
2. Собрать через `make` с использованием PGXS — на выходе получить `.so` и скопировать (`make install`) в каталоги, которые сервер знает по умолчанию.
3. Если расширению нужна разделяемая память, background worker или ранняя установка хуков — прописать библиотеку в `shared_preload_libraries` и перезапустить сервер.
4. Выполнить `CREATE EXTENSION имя;` в целевой базе данных — это создаст SQL-объекты расширения согласно `.sql`-скрипту, на который указывает `.control`-файл.
5. Дальнейшие вызовы функций расширения из SQL приводят к тому, что backend-процесс (если библиотека ещё не была предзагружена постмастером) динамически подгружает `.so` при первом обращении.

## Заключение

Архитектура PostgreSQL построена вокруг небольшого числа устойчивых идей: строгое разделение процессов (а не потоков) для изоляции сбоев, чёткое разделение frontend/backend-кода в дереве исходников, и максимально широкие возможности расширения функциональности снаружи через понятный C-API (хуки, GUC, регистрация фоновых процессов) без необходимости патчить ядро. Postmaster выступает единой точкой входа, которая на старте готовит разделяемые ресурсы (в том числе для расширений из `shared_preload_libraries`), а затем в цикле `ServerLoop` порождает и контролирует все остальные процессы кластера. Расширения вроде `pg_stat_statements` — не какое-то особое, "привилегированное" исключение, а демонстрация штатного публичного механизма, доступного любому разработчику: тот же набор макросов, хуков и файлов управления используется и внутренними, и сторонними модулями.
