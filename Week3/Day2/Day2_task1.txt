Идея: берём сценарий с высокой конкуренцией за одну и ту же горячую 
строку/spinlock — UPDATE counters SET value = value + 1 WHERE id = 1, 
запущенный сразу несколькими клиентами 
(pgbench -f update_hot_row.sql -c N -j N, где N ≥ числа физических ядер). 
ри включённом SMT два логических потока одного физического ядра одновременно 
пытаются менять одну и ту же строку/буфер, из-за чего растёт время удержания 
spinlock и cache-line bouncing между потоками — полезная работа на такт 
CPU падает. TPS снижается, потому что транзакции чаще стоят в очереди за 
блокировкой, а не выполняются параллельно. CPU utilization при этом тоже падает
(а не растёт), потому что часть логических CPU простаивает в 
busy-wait/спине на блокировке вместо реальных вычислений, и планировщик 
реже успевает выдать им полезную работу.
